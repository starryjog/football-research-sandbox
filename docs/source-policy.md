# 来源政策

本文定义数据来源优先级、交叉验证规则和冲突处理方式。所有新增或修改数据都应能追溯到公开来源。

## 来源优先级

从高到低优先考虑：

1. 官方原始来源：赛事方 PDF、协会公告、俱乐部官网、官方报名表、官方比赛报告、官方赛程和结果页。
2. 权威数据库：Transfermarkt、Soccerway、Soccerbase、FBref、FIFA 或 AFC 数据页等可回查数据库。
3. 主流媒体和地方媒体：有署名、发布时间、原文链接和明确事实描述的报道。
4. 专题、百科和社区汇总：Wikipedia、懂球帝、直播吧、论坛或二次整理文章，可用于线索和交叉验证，但不宜单独决定关键字段。
5. 视频、社媒和搜索结果：只适合作为旁证或入口，除非能稳定定位到官方账号、公开视频标题、发布日期和明确内容。

不同字段可使用不同来源。例如赛事报名优先官方 PDF，身价只作为 Transfermarkt 口径，球员学校或青训路径可能需要协会公告、学校新闻和媒体报道交叉确认。

## 最低核验要求

新增关键记录时，至少确认：

- 身份：姓名、出生日期或年龄段、国籍/协会、位置。
- 归属：注册俱乐部、学校、项目或国家队口径。
- 事件：赛事、赛季、报名、出场、转会、留洋注册或专题节点。
- 时间：事件日期、赛季或核验日期。
- 来源：公开 URL、来源标题和必要的备注。

球员条目应至少包含一个稳定外部链接。留洋、赛事和专题记录应尽量保留多个来源链接。

## 交叉验证

以下情况必须交叉验证：

- 中文名、英文名、日文名、韩文名或本地名存在多种写法。
- 同名球员可能混淆。
- 出生日期、位置、俱乐部、学校或国籍在不同来源不一致。
- 留洋记录涉及正式出场、注册状态、梯队/预备队/低级别联赛边界。
- 市场价值、出场次数、国家队场次等数字会被不同数据库按不同口径统计。

交叉验证不要求所有来源完全一致，但需要在备注中说明采用哪个口径。

## 冲突处理

发现冲突时：

1. 保留更接近事件发生方的来源。
2. 不删除仍有参考价值的冲突信息，除非确认是错误或重名误判。
3. 在 `verification.notes`、`notes`、`scope_note` 或专题说明中写明差异。
4. 避免把推断写成事实。
5. 给需要后续核验的问题留下明确 TODO 或 open question。

示例：

- AFC 报名表与俱乐部页面的当前俱乐部不同：报名表用于该赛事时点，俱乐部页面用于当前注册状态。
- Transfermarkt 与 Wikipedia 出场数不同：说明统计范围是联赛、各项赛事还是国家队。
- 媒体写“留洋”但没有注册或出场来源：先按观察或专题备注处理，不并入已验证留洋样本。

## 来源字段写法

来源链接至少包含：

- `label`：来源名称，能看出发布方和内容。
- `url`：公开可访问 URL。
- `publisher`：如当前文件已使用该字段，则补上发布方。

日期字段使用 `YYYY-MM-DD`。若来源只有月份或赛季，不要伪造精确日期；可以把精确日期留给核验日期，把事件时间写在摘要或备注中。

## Transfermarkt 与市场价值

Transfermarkt 可用于补充球员页面、注册路径和市场价值口径，但它不是项目的开放数据来源。相关数据应只作为外部来源引用和研究索引，不应把 Transfermarkt 的数据库内容重新授权为本站数据。

`data/raw/player-market-values.json` 记录的是站内快照。更新前应确认球员已有关联 Transfermarkt 个人页，并运行对应同步脚本。展示时必须保留来源口径，不把身价解释为官方估值。

## 未成年人和隐私

本项目只记录公开、足球相关、对研究必要的信息。不要新增联系方式、家庭信息、住址、身份证明、私人账号、未公开伤病或学校内部资料。

对未成年球员尤其应保持克制：优先记录赛事报名、公开俱乐部/学校归属和比赛路径，避免扩展到与足球研究无关的私人信息。

## 引用与授权

外部来源的文字、图片、名单、数据库字段和商标仍归原权利方所有。本项目的 `LICENSE` 不会重新授权这些内容。引用资料时尽量摘要化、结构化，并保留原始链接。
